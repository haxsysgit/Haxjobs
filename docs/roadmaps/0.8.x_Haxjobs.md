# HaxJobs 0.8.x Roadmap — Production Hardening and Polish

## Major theme

0.8.x makes HaxJobs reliable enough to use continuously.

By this point the main loop should exist:

```text
save job
→ analyze
→ generate pack
→ track application
→ collect profile answers
→ capture jobs from browser
→ assist apply/outreach with approval
```

0.8.x is about making that loop dependable, secure, deployable, and pleasant.

## Final outcome for 0.8.x

By the end of 0.8.x, HaxJobs should be closer to a real daily-use product:

1. Better reliability.
2. Better test coverage.
3. Cleaner deployment story.
4. Safer data handling.
5. Better profile/document export.
6. Stronger logs and debugging.
7. More polished UX.
8. Clear path toward 1.0.

## Product scope

Included:

- auth/user isolation if needed
- deployment setup
- Postgres migration path
- backups/export
- document cleanup/versioning
- stronger tests
- observability/logging
- UI polish
- extension packaging polish

Not included:

- enterprise CRM features
- mass auto-apply
- mass outreach
- huge scraping farm
- paid SaaS billing unless intentionally chosen later

# 0.8.0 — Test coverage hardening

## Goal

Make the system safer to change.

## Build

- backend unit tests
- backend integration tests
- frontend component tests where useful
- API contract tests
- task result writeback tests
- extension payload tests

## Done when

- core job/profile/task/pack workflows are covered by automated tests

## Goal

# 0.8.5 — Auth/user isolation decision

## Goal

Decide whether HaxJobs needs user accounts now.

## Options

Single-user local mode:

- simpler
- best for personal Hermes companion use

Multi-user mode:

- requires auth
- user_id on records
- stronger permissions
- more deployment complexity

## Done when

- product has an explicit auth/user-scope decision
- implementation matches that decision

---

# 0.8.6 — Security/privacy pass

## Goal

Review risky surfaces.

## Check

- no raw password storage
- sensitive answers labeled
- approval gates enforced

- extension permissions minimized
- CORS safe for intended deployment

- obvious security/privacy footguns are removed

---

# Risks and decisions

## Risk: polishing before the core loop proves useful

Only do this phase after the save/analyze/pack/profile/extension/apply-assist loop exists.

## Risk: turning into a heavy ATS/CRM

HaxJobs should stay a job-search command center, not enterprise software cosplay.

## Exit criteria for 0.8.x

0.8.x is complete when HaxJobs is reliable, safe, exportable, and polished enough for continuous real-world use.
