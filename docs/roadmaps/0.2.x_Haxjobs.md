# HaxJobs 0.2.x Roadmap — Hermes Task Queue and Job-Platform Foundation

## Major theme

0.2.x connects HaxJobs to Hermes conceptually and structurally.

The goal is to build the task queue and result writeback contract so HaxJobs can ask Hermes to do work and then store/display what Hermes produces.

This is the spine of the product.

## Final outcome for 0.2.x

By the end of 0.2.x, HaxJobs should be able to:

1. Create a HermesTask.
2. Store structured task input.
3. Track pending/running/completed/failed/blocked task states.
4. Accept structured Hermes results.
5. Update jobs/applications/packs/contacts from Hermes results.
6. Store source-platform metadata for common job boards.
7. Ingest raw snapshots ready for browser extension work.
8. Show enough task status for the UI to build on in 0.3.x.

## Product scope

Included:

- HermesTask lifecycle
- task input/result schema
- job source platform classification
- JobSourceSnapshot ingestion
- snapshot normalization flow
- polling-based task status
- optional SSE only if needed

Not included yet:

- full dashboard polish
- browser extension itself
- direct browser automation
- final application submission
- live multi-agent UI
- complex WebSocket protocol

# 0.2.0 — HermesTask queue contract

## Goal

Create the durable work queue that HaxJobs and Hermes share.

## Task lifecycle

```text
pending → running → needs_user_input → completed
                  ↘ failed
                  ↘ cancelled
```

## Build

- create task endpoint
- claim/start task endpoint or status update mechanism
- complete task endpoint
- fail task endpoint
- cancel task endpoint
- task list/filter endpoints

## Core fields

- task_type
- target_type
- target_id
- status
- instructions
- input_payload_json
- result_payload_json
- error
- created_at
- started_at
- completed_at

## Supported task types

- analyze_job
- generate_pack
- find_contacts
- draft_outreach
- apply_assist
- rank_saved_jobs
- refresh_application_status

Aliases that can be accepted early:

- find_contact → find_contacts
- draft_message → draft_outreach

## Done when

- HaxJobs can create a pending task
- Hermes can find pending tasks
- status updates persist
- failed tasks store useful error text
- task history is visible through API

# 0.2.1 — analyze_job task contract

## Goal

Define and implement the first useful Hermes task: analyze a job.

## Input

- job_id or snapshot_id
- job URL
- raw job description or visible page text
- source platform
- user profile summary

## Output

- normalized company
- normalized title
- normalized location
- normalized description
- fit_score
- strongest matching evidence
- major gaps
- sponsorship_risk
- recruiter priorities
- recommendation
- next_action
- questions if needed

## Writeback behavior

Hermes result should update:

- Job fields
- Application status
- fit_score
- sponsorship_risk
- recommendation
- next_action
- StatusEvent: job_analyzed

## Done when

- analyze_job result can turn a rough saved job/snapshot into a usable job detail record

# 0.2.2 — generate_pack task contract

## Goal

Prepare the system for Hermes-generated application packs.

## Input

- application_id
- job profile
- user profile facts
- saved answers
- previous similar packs if any
- generation mode

## Output

- fit summary
- tailored CV path
- cover letter path
- likely application Q&A path
- optional interview notes path
- optional combined pack path
- warnings/questions

## Writeback behavior

Hermes result should create/update:

- ApplicationPack
- Document records
- Application status: Pack Generated
- StatusEvent: pack_generated

## Done when

- HaxJobs can create a generate_pack task
- result can register generated files without manual copying

# 0.2.3 — find_contacts task contract

## Goal

Let Hermes search for relevant people, while HaxJobs stores candidates safely.

## Input

- company
- role title
- job URL
- source platform
- job poster info if available

## Output

- contact candidates
- names
- titles
- company
- platform
- profile URL
- email if publicly available/appropriate
- relevance reason
- confidence

## Writeback behavior

Hermes result should create:

- Contact records
- Application/Job timeline events

## Done when

- HaxJobs can store contacts from Hermes without sending any messages

# 0.2.4 — draft_outreach task contract

## Goal

Let Hermes draft natural outreach, but keep sending approval-gated.

## Input

- contact_id
- application_id
- job summary
- user profile angle
- pack/fit summary

## Output

- message text
- channel recommendation
- risks/notes

## Writeback behavior

Hermes result should create:

- OutreachMessage with status drafted
- StatusEvent: message_drafted

## Safety

No message should be sent from this task. Sending belongs later, behind approval checkpoints.

## Done when

- HaxJobs stores outreach drafts for user review

---

# 0.2.5 — apply_assist task contract

## Goal

Define the shape for apply assistance without making it a main product feature yet.

## Input

- job/application
- pack files
- saved answers
- login/session availability
- approval checkpoint references if any

## Output

- fields completed
- files uploaded
- blockers/questions
- final review summary
- submitted status only if explicitly approved and actually submitted

## Writeback behavior

Possible updates:

- Application status: Applying / Needs User Input / Applied
- ApprovalCheckpoint requirement
- StatusEvent: application_started, needs_user_input, application_submitted

## Done when

- the contract exists
- real apply automation can wait until 0.7.x

# 0.2.6 — Job platform source classification

## Goal

Normalize where jobs come from.

## Initial source platforms

- linkedin
- indeed
- reed
- workday
- greenhouse
- lever
- ashby
- company_site
- manual
- other

## URL rules

```text
linkedin.com/jobs → linkedin
indeed.com → indeed
reed.co.uk → reed
myworkdayjobs.com → workday
greenhouse.io → greenhouse
lever.co → lever
ashbyhq.com → ashby
otherwise → company_site or other
```

## Done when

- HaxJobs can classify source URLs consistently
- source_platform is stored on Job and JobSourceSnapshot

# 0.2.7 — JobSourceSnapshot ingestion

## Goal

Prepare for browser extensions by accepting raw page captures.

## Input payload

- url
- title
- source_platform guess
- visible_text
- selected_text
- user_note
- captured_at

## Behavior

- save raw snapshot

## Done when

# 0.2.8 — Task status updates: polling first

## Goal

Let the frontend observe task state without overbuilding realtime infrastructure.

## Build

- task list polling endpoint
- task detail endpoint
- updated_at timestamp
- filters by status/type/target

## Default decision

Use polling first.

Upgrade path:

1. polling
2. Server-Sent Events
3. WebSockets only if bidirectional live interaction is required

## Done when

- frontend can refresh task state predictably
- no WebSocket dependency exists yet

# Risks and decisions

## Risk: HaxJobs accidentally becomes the agent

Do not put reasoning/automation into HaxJobs. HaxJobs creates/stores tasks. Hermes performs tasks.

## Risk: task results too free-form

Hermes output should include summaries for humans, but also structured fields for UI writeback.

## Risk: realtime rabbit hole

Do not build WebSockets just because they sound nice. Polling is fine for MVP.

## Exit criteria for 0.2.x

0.2.x is complete when HaxJobs can create Hermes tasks, accept structured results, update durable state, and ingest job snapshots ready for extension capture.
