# HaxJobs 0.1.x Roadmap — Foundation and Local Product Skeleton

## Major theme

0.1.x is the foundation line.

The goal is not to build every product feature yet. The goal is to make HaxJobs structurally real:

- backend exists
- frontend exists
- database exists
- core entities exist
- local persistence works
- manual job save works
- the repo has a clean shape for later Hermes integration

Think of this as pouring the concrete before building the house. Boring, but if this part is weak, everything later becomes pain.

## Final outcome for 0.1.x

By the end of 0.1.x, a user should be able to:

1. Run HaxJobs locally.
2. Open the web UI.
3. Save a job manually.
4. See the saved job in persistent storage.
5. Track a basic application record/status.
6. Store the first version of user profile data.
7. Attach/register local document paths.
8. Trust that the database model can support later Hermes tasks, packs, contacts, and approvals.

## Product scope

Included:

- project setup
- local development workflow
- backend/frontend skeleton
- database and migrations
- core data model
- core CRUD APIs
- manual job creation
- status-event timeline foundation
- basic local file/document storage

Not included yet:

- browser extension
- real Hermes worker execution
- automated job-board scraping
- polished dashboard
- generated PDFs
- apply automation
- outreach automation
- WebSockets/realtime complexity

## Architecture direction

Recommended stack:

- Backend: Python + FastAPI
- Frontend: Vue 3 + Vite
- Database: SQLite first
- Future production database: PostgreSQL
- ORM/migrations: SQLAlchemy + Alembic, or SQLModel + Alembic
- Document storage: local filesystem first
- Agent integration: schema-ready only in 0.1.x, real queue in 0.2.x

Suggested repo shape:

```text
haxjobs/
  backend/
    haxjobs_api/
      main.py
      config.py
      database.py
      models/
      schemas/
      api/
      services/
    tests/
    alembic/
  frontend/
    src/
      views/
      components/
      services/
      stores/
  docs/
    ROADMAP.md
    roadmaps/
  data/
    .gitkeep
    documents/
```

Keep separation clean:

- API routes should not contain business logic.
- Services should own workflow decisions.
- Models should represent storage.
- Schemas should represent API input/output.
- Frontend services should own HTTP calls.

# 0.1.1 — Database setup and migrations

## Goal

Add durable local persistence with migrations from the start.

## Build

- SQLite database configuration
- database session/dependency
- Alembic migration setup
- base model conventions
- timestamp fields
- ID strategy

## Design choices

Use SQLite locally, but avoid SQLite-only assumptions where easy.

Recommended:

- UUID/string IDs, or integer IDs with clear API boundaries
- timezone-aware timestamps
- JSON fields where useful, but keep core searchable fields as columns

## Done when

- migration command creates database tables
- test database can be created independently
- app can open a DB session in tests

# Risks and decisions

## Risk: overbuilding too early

Do not build advanced automation in 0.1.x. If Hermes integration appears, it should be schema preparation only.

## Risk: weak data model

The model should support later features, but do not make it too abstract. The entities in this roadmap are enough.

## Decision: SQLite first

SQLite is enough locally. Keep Postgres in mind, but do not block MVP on infrastructure.

## Decision: local files first

Generated packs can be local files first. Cloud storage can wait.

## Exit criteria for 0.1.x

0.1.x is complete when HaxJobs is a working local CRUD app with the correct core data model and manual job tracking.
