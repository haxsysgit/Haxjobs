# HaxJobs Harness Plan

## Scope

Design the HaxJobs harness only.

- Optimize for a target architecture, not just a patch on the current loop.
- Specify the full end-state harness shape now.
- Keep business logic in shared product actions outside the harness.

## Ground truth from docs and code

- The harness is supposed to stay thin: prompt assembly, tool exposure, tool dispatch, result appending, loop control, and approval-safe orchestration.
- The current implementation in `src/haxjobs/agent/agent.py` is still a short in-memory loop with no durable session or run model.
- `src/haxjobs/agent/registry.py` is already a good small registry/dispatch boundary and should stay narrow.
- `src/haxjobs/agent/prompt.py` and `src/haxjobs/agent/prompts.py` represent a transitional prompt system and should converge into one canonical prompt path.
- `docs/HAXJOBS_AGENT_HARNESS.md` and `docs/ROADMAP.md` both point to the same missing fundamentals: durable sessions, runs, checkpoints, retrieval/context assembly, compaction, telemetry, child-run isolation, resume, and stop controls.

## Recommended architecture

Build the harness as a small durable runtime around shared product tools.

### 1. Public facade

Keep `src/haxjobs/agent/agent.py` as the public entrypoint only.

- It should build a run request.
- It should call the runtime.
- It should return the final assistant result.
- It should stop owning the full tool loop directly.

### 2. Runtime lifecycle

Introduce one canonical runtime layer inside `src/haxjobs/agent/`.

Responsibilities:

- create or resume a session/run
- build model input from prompt layers and stored context
- execute the model/tool loop
- persist messages and tool events
- enforce limits, stop reasons, and approval pauses
- checkpoint after meaningful progress

This is the main missing foundation and should be treated as one design, not a pile of helpers.

### 3. Tool policy boundary

Evolve `src/haxjobs/agent/tool_modes.py` into an explicit run policy layer.

Responsibilities:

- mode-to-tool mapping
- allowlists and deny lists
- admin-only visibility
- approval-required metadata
- child-run restricted tool views

The key change is that tool visibility becomes persisted run configuration, not just a convenience flag.

### 4. Prompt stack

Unify prompt construction under one canonical path.

Recommended shape:

- stable identity and safety prefix from `src/haxjobs/agent/identity.py`
- flow instructions for the current run mode
- selected harness checkpoint summary
- minimal recent raw history
- current volatile task context

`src/haxjobs/agent/prompts.py` should either become a library of reusable flow templates used by `prompt.py`, or be absorbed into a single prompt builder. No parallel prompt systems.

### 5. Durable harness state

The harness should own only the runtime state needed for resumability and observability.

Minimum entities:

- session
- run
- message
- tool event
- checkpoint or compaction record
- usage ledger

Recommended persisted fields:

- session: id, status, timestamps
- run: id, session id, parent run id, mode, model, policy snapshot, status, stop reason, timestamps
- message: id, run id, role, content, turn index, source type
- tool event: tool name, args JSON, result JSON, success or failure, latency, timestamp
- checkpoint: covered message range, summary text, summary kind
- usage ledger: tokens, cached tokens when available, latency, estimated cost

Use SQLite, but keep harness tables logically separate from product tables.

### 6. Context assembly and compaction

Add a dedicated layer that decides what goes into each model call.

Responsibilities:

- select the latest checkpoint summary
- include only the minimum recent raw turns
- inject explicit volatile context
- track context size and trimming decisions
- compact older history into stored summaries

The harness should retrieve from harness state first, not by rebuilding context ad hoc every time.

### 7. Approval and child runs

Add these as runtime features, not business features.

Approval layer responsibilities:

- detect approval-sensitive actions
- pause the run cleanly
- persist pending approval state
- resume without losing run context

Child run responsibilities:

- isolated sub-run with explicit objective
- restricted prompt and restricted tool policy
- bounded lifecycle
- explicit result returned to the parent run

## Target module shape

Converge toward this layout under `src/haxjobs/agent/`:

- `agent.py` - public facade only
- `runtime.py` - main run lifecycle
- `sessions.py` - session, run, message persistence API
- `context.py` - context selection and prompt assembly inputs
- `checkpoints.py` - compaction and checkpoint summaries
- `telemetry.py` - token, latency, cost, and error recording
- `policy.py` - tool policy and approval metadata
- `registry.py` - tool registration and dispatch only
- `prompt.py` - canonical prompt builder
- `identity.py` - identity and stable safety text
- `child_runs.py` - isolated sub-run contract
- existing `tools_*.py` modules remain thin typed adapters

This shape is the destination. It does not all need to land in one implementation slice.

## Execution phases

### Phase 1. Normalize the harness contract

- Make `agent.py` a facade conceptually and then structurally.
- Freeze `registry.py` as registration and dispatch only.
- Decide on one prompt system and remove the parallel prompt path.
- Turn tool mode selection into explicit run configuration.

### Phase 2. Add durable runs and traces

- Persist sessions, runs, messages, and tool events.
- Record run status, stop reason, and failures.
- Add the minimum resume path for interrupted runs.

### Phase 3. Add context assembly and checkpoints

- Add checkpoint summaries and compaction records.
- Assemble prompts from stable prefix, flow instructions, checkpoint summary, recent raw history, and live context.
- Add token and latency tracking.

### Phase 4. Add approval pause/resume and child-run isolation

- Introduce explicit approval wait state.
- Add isolated child runs with restricted policies.
- Add stronger stop conditions and verification hooks.

### Phase 5. Expose the runtime cleanly

- Keep `haxjobs agent ask` as a thin surface over the runtime.
- Prepare the harness so future CLI and API callers can reuse the same runtime without duplicating loop logic.

## Decisions to lock now

- The harness owns runtime mechanics only, not career workflow logic.
- There will be one canonical prompt assembly path.
- Run configuration must persist mode, model, and tool policy snapshot.
- Checkpoints and compaction summaries are first-class stored artifacts.
- Approval waits are a run state, not a special-case error.
- Child runs are isolated by default.

## Decisions to defer

- exact retrieval and ranking strategy
- exact compaction thresholds
- exact cost accounting details across providers
- richer child-run merge behavior
- broader product memory model beyond what the harness itself needs

## Verification

Implementation should be verified at three levels.

### Unit and integration

- Add focused tests for run creation, tool dispatch recording, resume behavior, checkpoint creation, context assembly, approval pause/resume, and child-run isolation.
- Keep `registry.py` tests narrow and deterministic.
- Add prompt builder tests that prove stable versus volatile sections are assembled consistently.

### CLI and runtime behavior

- Verify `haxjobs agent ask` still works for plain single-turn text.
- Verify a tool-enabled run records messages and tool events durably.
- Verify an interrupted run can resume.
- Verify checkpointed long runs still produce correct prompts after compaction.

### Repo-level checks

Run the standard repo verification from `AGENTS.md` after implementation:

- `PYTHONPATH=src:. uv run python3 -m pytest -q tests/`
- `PYTHONPATH=src:. uv run python3 -m py_compile $(find src tests cron -name '*.py')`
- `bash -n cron/run_pipeline.sh`
- frontend checks only if harness changes require touching shared web surfaces:
  - `cd frontend`
  - `npx tsc --noEmit`
  - `npm run lint -- --quiet`
  - `npm run build`

## Recommendation

The first real implementation slice should be Phase 1 plus Phase 2: one canonical prompt/runtime contract and durable session/run/tool-event storage. That gives HaxJobs a real harness foundation without pulling business logic into the loop.
