## 1. Executive Summary

**The pipeline brain works. The dashboard body is decorative.**

The discovery → evaluation → pack generation pipeline is functional. Jobs get scraped, filtered, evaluated, and scored. The enriching we just did (whitelist, company notes, behavioral guardrails) makes the evaluation smarter. Packs get generated for strong fits.

But the dashboard — the thing you actually use to control this system — is mostly read-only, hardcoded, or stubbed. You cannot do basic things like view why a job was skipped, manually approve a job, manage packs, or toggle auto-apply. The frontend doesn't even call the right API endpoints for the features that DO exist on the backend.

**Verdict: The stack is fine. The architecture needs separation of concerns. The UI needs a complete rebuild of its interactive layer.**

## 2. Real Bugs Found (not theoretical)

### BUG-1: Unskip button creates a duplicate job instead of unskipping
- **File:** `dashboard/src/App.tsx:53-63` (handleUnskip)
- **What happens:** Calls `api.queueIntake()` which creates a NEW intake job. It does NOT call `api.unskipJob()` which hits `POST /api/jobs/unskip`.
- **Result:** Clicking "Unskip -> Evaluate" on a skipped job creates a duplicate entry that gets evaluated AGAIN with the same old prompt (since it's a new intake, not the original job with fresh status). The original skipped job stays in the filtered list forever.
- **Fix:** Change `handleUnskip` to call `api.unskipJob(job.id, reason)` and then immediately refresh the jobs list locally.

### BUG-2: Filtered list doesn't update after unskip
- **File:** `dashboard/src/pages/Pipeline.tsx:113`
- **What happens:** `filtered = jobs.filter(j => j.status === 'skipped' || ...)`. When a job is unskipped (status changes to 'pending'), it should disappear from Filtered and appear in Pending. But the jobs array in App.tsx state only refreshes every 30 seconds via polling, and there's no optimistic update.
- **Fix:** After unskip/approve, immediately update local state rather than waiting for the next poll.

### BUG-3: Cannot view details of skipped or pending jobs
- **File:** `dashboard/src/pages/Pipeline.tsx:57-67` (PendingList), `:69-97` (FilteredList)
- **What happens:** Only the Evaluated tab jobs have `onClick` handlers that open JobDetail. PendingList cards are plain divs with no click handler. FilteredList cards have `cursor: 'default'` and no `onSelect` prop passed at all.
- **Result:** You cannot click on any skipped job to see WHY it was skipped. You cannot click on pending jobs to see their raw JD. The only way to inspect them is to SSH into the VPS and read the intake JSON files.
- **Fix:** Pass `onSelect` to both PendingList and FilteredList. Make JobDetail handle non-evaluated jobs gracefully (show raw JD, show skip_reason prominently).

### BUG-4: JobDetail pack links are all dead stubs
- **File:** `dashboard/src/pages/JobDetail.tsx:62-76`
- **What happens:** Four hardcoded links to PDFs — all have `href="#" onClick={e => e.preventDefault()}`. They do nothing. There's no API endpoint to serve pack files, no pack browser, no way to download a generated CV from the dashboard.
- **Fix:** Either serve pack files via the API server (add a `/api/packs/:id/files/:filename` route), or generate a packs browser page.

### BUG-5: api.ts client missing half the backend endpoints
- **File:** `dashboard/src/data/api.ts`
- **What exists on backend but NOT in frontend client:**
  - `POST /api/jobs/unskip` — exists but frontend uses wrong endpoint (BUG-1)
  - `POST /api/jobs/approve` — exists, no frontend method at all
  - `GET /api/whitelist` — exists, no frontend method
  - `POST /api/whitelist` — exists, no frontend method
  - `POST /api/whitelist/remove` — exists, no frontend method
  - `GET /api/packs` — exists, no frontend method
  - `POST /api/saved-jobs` — exists, no frontend method
  - `POST /api/saved-jobs/remove` — exists, no frontend method

### BUG-6: Activity log strips all metadata
- **File:** `dashboard/src/pages/Activity.tsx:12`
- **What happens:** `data.map((l: any) => l.message || '').filter(Boolean)` — throws away timestamps, event types, job IDs. You just see raw message strings with no context.
- **Fix:** Show timestamp + type + message, make jobs clickable.

## 4. Architecture Audit

### 4.1 What's working well

| Component | Assessment |
|---|---|
| **Discovery scrapers** | Solid. Lever/Ashby/Greenhouse API scrapers work reliably. Sharp filter catches non-engineering roles. UK location filter prevents 76% waste. |
| **Evaluation pipeline** | Good. Single-job evaluation via Hermes is the right approach. Enriched profile + whitelist + company notes now makes it smarter. |
| **SQLite DB** | Good choice for this scale. WAL mode, foreign keys, proper schema. No need for Postgres. |
| **System crontab** | Correct decision. Shell scrapers don't need LLM tokens. |
| **React + Vite** | Fine. Modern, fast builds. No issues with the framework itself. |

### 4.2 What needs restructuring

| Component | Problem | Recommendation |
|---|---|---|
| **api_server.py (427 lines)** | Monolithic: routing + static serving + JSON parsing + CORS + business logic all in one file | Split into: `server.py` (entry), `routes/` (endpoints), `services/` (business logic), `middleware.py` (CORS/logging) |
| **pipeline_db.py (617 lines)** | Schema + CRUD + stats + CLI + seeding + whitelist all in one file | Split into: `db/schema.py`, `db/jobs.py`, `db/evaluations.py`, `db/whitelist.py`, `db/stats.py` |
| **App.tsx** | All state at app level, 30-second poll, no optimistic updates | Add TanStack Query or a simple custom hook for per-page data fetching with cache invalidation on mutations |
| **No React Router** | `useState('dashboard')` for navigation. No URLs, no back button, no deep links | Add React Router. Pages become real URLs. |
| **Dual storage (DB + JSON)** | Intake JSON files AND SQLite rows for the same data. Creates sync issues. | Make DB the single source of truth. Intake JSON becomes write-only audit trail, never read back for state. |
| **No proper error handling** | `try/catch { /* offline */ }` everywhere. No error states in UI. | Add error boundaries, toast notifications, retry buttons. |

### 4.3 Stack assessment

**Current stack is NOT wrong.** React + TypeScript + SQLite + Python is perfectly fine for this scale. The problems are architectural (monolithic files, no separation of concerns) and feature-completeness (UI is decorative), not stack-choice problems.

**What I would change:**
- Backend: Move from `http.server` to **FastAPI**. Auto-generated docs alone would let you see every endpoint. Pydantic validation would catch bugs early. But this is a nice-to-have, not urgent.
- Frontend: Add **React Router** (deep linking) and **TanStack Query** (proper data fetching). Both are 5KB gzipped. Low cost, high impact.
- Keep: SQLite, system crontab, Python for pipeline, React + Vite for dashboard.

**What I would NOT change:**
- Do NOT add Postgres. SQLite handles 108 jobs perfectly.
- Do NOT add Redux/Zustand. TanStack Query handles server state. Local state is fine for UI.
- Do NOT migrate to Next.js. SPA served by Python API is fine. No SSR needed.
- Do NOT add Docker. Systemd + crontab works.

## 7. Design Decisions I'm Making Without Asking You

These are things I'm confident about from reading every line of code:

1. **DB is the single source of truth.** Intake JSON files become write-only audit trail. No more reading back from files. This eliminates the dual-format bug (fit_score vs fit_report nesting).

2. **Unskip means "re-evaluate with enriched profile."** Not "create a duplicate." The enriched prompt now includes whitelist + company notes, so re-evaluation should produce a different result.

3. **Approve means "bypass evaluation entirely, ready for pack generation."** This is already implemented in the backend. Just needs a UI button.

4. **Auto-apply is a flag, not an action.** Marking a job as "auto-apply" means the pipeline will submit it (with your approval checkpoint). The actual submission still goes through you.

5. **The current stack stays.** FastAPI migration is a Phase 4 nice-to-have, not a blocker. The current `http.server` works and we should fix the functional gaps before refactoring the transport layer.

6. **React Router added.** No more `useState('dashboard')` navigation. Real URLs, browser back button works.
